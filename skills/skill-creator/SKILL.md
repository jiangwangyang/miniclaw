# 技能创建工具

本技能为高效技能的创建提供指导方案。

## 关于技能

技能是**模块化、独立封装**的程序包，通过提供专业知识、工作流与工具来扩展 Agent 的能力。可以将其视为特定领域或任务的「上手指南」—— 它们能将 Agent 从一个通用智能体，转变为具备「过程性知识」的专业智能体，而这类知识是任何模型都无法完全内置的。

### 技能的核心价值

1. **专业工作流** — 面向特定领域的多步骤操作流程
2. **工具集成能力** — 针对特定文件格式或 API 的使用说明
3. **领域专业知识** — 企业专属知识、数据架构、业务逻辑
4. **打包资源** — 用于处理复杂重复性任务的脚本、参考文档与资源文件

## 核心原则

### 简洁至上

上下文窗口是共享资源。技能的内容需要与 Agent 所需的其他所有内容共用上下文窗口，包括系统提示词、对话历史、其他技能的元数据，以及用户的实际请求。

**默认前提**：Agent 本身已具备较强的智能水平。**仅添加 Agent 不具备的上下文信息**。对每一条信息都要审慎考量：「Agent 真的需要这段解释吗？」「这段内容的 token 消耗是否有必要？」

优先使用简洁示例，而非冗长说明。

### 合理设定自由度

根据任务的**容错性**与**可变性**，匹配对应的规范粒度：

- **高自由度（基于文本的指令）**：适用于存在多种可行方案、决策依赖上下文，或需依靠经验法则指导的场景。
- **中自由度（带参数的伪代码/脚本）**：适用于存在优选范式、允许一定灵活度，或配置会影响执行行为的场景。
- **低自由度（固定脚本+少量参数）**：适用于操作容错率低、易出错，一致性要求极高，或必须严格遵循特定步骤的场景。

可以这样理解：把 Agent 比作在路径上探索的智能体 —— 悬崖边的窄桥需要明确的防护栏（低自由度），而开阔的平原则允许自由选择路线（高自由度）。

### 技能的结构组成

每个技能都包含一个**必填**的 `SKILL.md` 文件，以及**可选**的打包资源，目录结构如下：

```
技能名称/
├── SKILL.md （必填）
│   ├── YAML 头部元数据 （必填）
│   │   ├── name: （必填）
│   │   └── description: （必填）
│   └── Markdown 格式的使用说明 （必填）
└── 打包资源 （可选）
    ├── scripts/          - 可执行代码（Python/Bash 等语言）
    ├── references/       - 按需加载到上下文的参考文档
    └── assets/           - 输出过程中用到的文件（模板、图标、字体等）
```

#### SKILL.md 文件（必填）

所有 `SKILL.md` 文件均包含两部分：

1. **头部信息（YAML 格式）**：包含 `name`（名称）和 `description`（描述）两个字段。这两个字段是 Agent 判断是否启用该技能的唯一依据，因此清晰、全面地描述技能的功能与适用场景至关重要。
2. **正文内容（Markdown 格式）**：技能的使用说明与操作指南。**仅在技能被触发后才会加载**。

#### 打包资源（可选）

##### 脚本目录 (`scripts/`)

存放用于实现确定性任务，或需重复编写的可执行代码（Python/Bash 等语言）。

- **适用场景**：需要重复编写相同代码，或对执行结果的确定性要求高时。
- **示例**：存放 `scripts/rotate_pdf.py` 脚本，用于处理 PDF 旋转任务。
- **优势**：节省 token 消耗、执行结果稳定，部分脚本无需加载到上下文即可直接运行。
- **注意事项**：脚本仍可能需要被 Agent 读取，以进行补丁修复或适配特定运行环境。

##### 参考文档目录 (`references/`)

存放用于指导 Agent 执行流程的参考文档与资料，可按需加载到上下文。

- **适用场景**：需要 Agent 在执行任务时参考特定文档的场景。
- **示例**：`references/finance.md`（财务数据架构）、`references/mnda.md`（企业保密协议模板）、`references/policies.md`（企业制度）、`references/api_docs.md`（API 接口文档）。
- **典型用途**：存储数据库架构、API 文档、领域知识、企业制度、详细工作流指南。
- **优势**：保持 `SKILL.md` 文件简洁，仅在 Agent 需要时才加载对应内容。
- **最佳实践**：若文件篇幅过长（超过 1 万字），需在 `SKILL.md` 中添加 grep 搜索规则。
- **避免重复**：信息应仅存于 `SKILL.md` 或参考文档中的一处。非核心的详细信息优先放入参考文档 —— 这既能保持 `SKILL.md` 精简，又能确保信息可被按需检索。仅将核心流程说明与工作流指南保留在 `SKILL.md` 中，把详细参考资料、数据架构与示例移至参考文档。

##### 资源文件目录 (`assets/`)

存放不加载到上下文，但会被 Agent 用于生成最终输出的文件。

- **适用场景**：技能需要调用特定文件来生成最终输出结果时。
- **示例**：`assets/logo.png`（品牌标识）、`assets/slides.pptx`（PPT 模板）、`assets/frontend-template/`（HTML/React 项目脚手架）、`assets/font.ttf`（字体文件）。
- **典型用途**：存储模板文件、图片、图标、项目脚手架代码、字体、可复用的示例文档。
- **优势**：将输出资源与参考文档分离，Agent 无需加载文件内容即可直接调用。

#### 技能中不应包含的内容

技能应仅包含直接支撑其功能的核心文件。**禁止**添加无关的说明文档或辅助文件，包括但不限于：

- README.md
- INSTALLATION_GUIDE.md
- QUICK_REFERENCE.md
- CHANGELOG.md
- 其他类似文件

技能的作用是为 AI 智能体提供完成任务所需的信息，不应包含技能开发过程说明、安装测试步骤、面向终端用户的使用文档等额外内容。多余的文档只会造成目录混乱，影响使用效率。

### 渐进式信息披露设计原则

技能采用三级加载机制，以实现上下文的高效管理：

1. **元数据（名称 + 描述）** — 始终处于上下文窗口中（约 100 词）
2. **SKILL.md 正文** — 技能被触发时加载（不超过 5000 词）
3. **打包资源** — Agent 按需加载（无字数限制，因为脚本可直接执行而无需加载到上下文）

#### 渐进式信息披露的实现模式

将 `SKILL.md` 正文内容控制在核心必要范围内，篇幅不超过 500 行，避免占用过多上下文空间。当内容接近此上限时，拆分到独立文件中。拆分内容时，**必须在 `SKILL.md` 中添加引用链接，并明确说明该文件的加载场景**，确保技能使用者知晓文件的存在及适用时机。

**核心原则**：当技能支持多种变体、框架或选项时，仅在 `SKILL.md` 中保留核心工作流与选型指引，将变体相关的细节（范式、示例、配置）移至独立的参考文档。

**模式一：顶层指南 + 参考文档**

```markdown
# PDF 处理技能

## 快速入门

使用 pdfplumber 提取文本：
[代码示例]

## 高级功能

- **表单填写**：完整指南请查看 [FORMS.md](FORMS.md)
- **API 参考**：所有方法请查看 [REFERENCE.md](REFERENCE.md)
- **使用示例**：常见场景请查看 [EXAMPLES.md](EXAMPLES.md)
```

Agent 仅在需要时才加载 `FORMS.md`、`REFERENCE.md` 或 `EXAMPLES.md`。

**模式二：按领域组织内容**
对于覆盖多领域的技能，按领域划分内容，避免加载无关上下文：

```
bigquery-skill/
├── SKILL.md （总览与导航）
└── reference/
    ├── finance.md （收入、计费指标）
    ├── sales.md （销售机会、销售漏斗）
    ├── product.md （API 用法、功能说明）
    └── marketing.md （营销活动、归因分析）
```

当用户询问销售指标相关问题时，Agent 仅加载 `sales.md`。

同理，对于支持多框架/多变体的技能，按变体划分内容：

```
cloud-deploy/
├── SKILL.md （工作流 + 云服务商选型指引）
└── references/
    ├── aws.md （AWS 部署范式）
    ├── gcp.md （GCP 部署范式）
    └── azure.md （Azure 部署范式）
```

当用户选择 AWS 时，Agent 仅加载 `aws.md`。

**模式三：条件式内容展示**
展示基础内容，将高级内容通过链接引出：

```markdown
# DOCX 文档处理技能

## 创建文档

使用 docx-js 库创建新文档，详情请查看 [DOCX-JS.md](DOCX-JS.md)。

## 编辑文档

简单编辑可直接修改 XML 结构。

**修订跟踪功能**：请查看 [REDLINING.md](REDLINING.md)
**OOXML 格式细节**：请查看 [OOXML.md](OOXML.md)
```

Agent 仅在用户需要对应功能时，才加载 `REDLINING.md` 或 `OOXML.md`。

**重要规范**

- **避免深层嵌套引用** — 所有参考文档应直接从 `SKILL.md` 引用，保持一级引用深度。
- **结构化长文档** — 对于超过 100 行的参考文档，需在顶部添加目录，方便 Agent 预览文件完整结构。

## 技能创建流程

技能创建需遵循以下步骤，仅当有明确理由时可跳过对应步骤：

1. 通过具体示例明确技能需求
2. 规划可复用的技能内容（脚本、参考文档、资源文件）
3. 编辑技能内容（实现资源文件并编写 `SKILL.md`）

### 技能命名规范

1. 仅使用小写字母、数字和连字符；用户提供的名称需标准化为连字符分隔格式（例如："Plan Mode" → `plan-mode`）。
2. 名称长度控制在 64 字符以内（仅限字母、数字、连字符）。
3. 优先使用简短的、动词开头的短语，精准描述技能功能。
4. 当有助于提升清晰度或触发准确性时，按工具名划分命名空间（例如：`gh-address-comments`、`linear-address-issue`）。
5. 技能文件夹名称必须与技能名称完全一致。

### 步骤 1：通过具体示例明确技能需求

仅当技能的使用场景已完全明确时，可跳过本步骤。即使是优化现有技能，本步骤依然具有参考价值。

要创建高效的技能，需先明确技能的具体使用场景示例。这些示例可来源于用户的实际需求，或通过生成示例并结合用户反馈验证得到。

例如，开发「图像编辑」技能时，可通过以下问题明确需求：

- 「图像编辑技能应支持哪些功能？比如编辑、旋转，或其他功能？」
- 「能否举例说明该技能的使用场景？」
- 「我猜测用户可能会提出‘去除图片红眼’‘旋转图片’这类需求，你认为还有其他使用场景吗？」
- 「用户输入哪些指令时，应该触发该技能？」

为避免给用户造成负担，**不要在单条消息中提出过多问题**。优先询问核心问题，再根据需求补充提问，以提升沟通效率。

当技能需支持的功能范围明确后，即可结束本步骤。

### 步骤 2：规划可复用的技能内容

要将具体示例转化为实用技能，需对每个示例进行如下分析：

1. 思考从零实现该示例所需的步骤
2. 识别在重复执行该工作流时，哪些脚本、参考文档和资源文件会有帮助

**示例 1**：开发 `pdf-editor` 技能以处理「帮我旋转这个 PDF」这类请求
分析结论：

1. 旋转 PDF 需要重复编写相同的代码
2. 可在技能中存放 `scripts/rotate_pdf.py` 脚本

**示例 2**：开发 `frontend-webapp-builder` 技能以处理「帮我搭建一个待办事项应用」「帮我开发一个步数统计仪表盘」这类请求
分析结论：

1. 开发前端应用需要重复编写相同的 HTML/React 脚手架代码
2. 可在技能中存放 `assets/hello-world/` 模板，包含 HTML/React 项目基础文件

**示例 3**：开发 `big-query` 技能以处理「今天有多少用户登录？」这类请求
分析结论：

1. 查询 BigQuery 需反复查阅数据表结构与关联关系
2. 可在技能中存放 `references/schema.md` 文件，记录数据表结构

通过对每个具体示例的分析，梳理出技能需包含的可复用资源清单，包括脚本、参考文档和资源文件。

### 步骤 3：编辑技能内容

编辑新生成或现有技能时，请牢记：技能是为其他 Agent 实例设计的工具。需包含对 Agent 有价值且非显而易见的信息，思考哪些过程性知识、领域细节或可复用资源，能帮助其他 Agent 实例更高效地完成任务。

#### 学习成熟的设计范式

可根据技能需求，参考以下指南文档：

- **多步骤流程**：参考 `references/workflows.md`，学习时序工作流与条件逻辑的设计方法
- **特定输出格式/质量标准**：参考 `references/output-patterns.md`，学习模板与示例的设计范式

这些文档包含了技能设计的最佳实践方案。

#### 从可复用资源开始实现

技能开发应从步骤 2 中梳理的可复用资源入手，优先完成 `scripts/`、`references/` 和 `assets/` 目录下的文件。注意，本步骤可能需要用户提供输入。例如，开发 `brand-guidelines`（品牌规范）技能时，可能需要用户提供品牌资源、模板文件（存入 `assets/`）或品牌规范文档（存入 `references/`）。

添加的脚本**必须经过实际运行测试**，确保无 bug 且输出结果符合预期。若存在大量相似脚本，可仅测试代表性样本，在保证可靠性的同时平衡开发效率。

若使用了 `--examples` 参数，删除技能不需要的占位示例文件。仅创建实际需要的资源目录。

#### 更新 SKILL.md 文件

**编写规范**：正文内容需使用**祈使句式**。

##### 头部信息编写

编写包含 `name` 和 `description` 的 YAML 头部信息：

- `name`：填写技能名称
- `description`：这是技能触发的核心依据，需清晰说明技能功能与适用场景
    - 同时包含技能的功能和触发条件/上下文
    - 所有「适用场景」信息均需写在此处 —— 不要写在正文中。正文仅在技能触发后加载，因此正文中的「技能适用场景」章节对 Agent 无效。
    - 示例（`docx` 技能的描述）：`提供全面的文档创建、编辑与分析能力，支持修订跟踪、批注添加、格式保留与文本提取。当 Agent 需要处理专业文档（.docx 文件）时启用，适用场景包括：(1) 创建新文档 (2) 修改或编辑内容 (3) 处理修订记录 (4) 添加批注，以及其他各类文档处理任务`

确保 YAML 头部信息格式合法，`name` 和 `description` 均使用单行字符串。若内容包含 YAML 特殊语法，需用引号包裹。

头部信息中**禁止添加其他字段**。

##### 正文内容编写

编写技能的使用说明，以及打包资源的调用指南。
